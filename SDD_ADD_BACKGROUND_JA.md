\# スペック駆動・エージェント駆動開発：背景と文脈



\---



\## 1. 起点：LLMをコーディングに使い始めた時期の問題



2022〜2023年、GitHub CopilotやChatGPTによるコード生成が普及した段階では、開発スタイルは「プロンプトを書いて返ってきたコードを貼る」という形式だった。



2025年2月、Andrej Karpathy（OpenAI共同創設者）がこの開発スタイルを「\*\*vibe coding\*\*」と命名した。「コードの存在を忘れ、ノリに完全に身を委ねる」という表現で、LLMへの高レベルな指示だけでコードを生成・修正させる手法を指す。



しかしこのアプローチには構造的な問題が伴った。



LLMベースのコーディングエージェントが2024〜2025年に主流になると、3種類の障害モードが顕在化した。第一は\*\*インテントドリフト\*\*：「ログイン追加」のような指示は過小仕様で、モデルが選ぶデフォルト値がチームの意図と合わないケース。第二は\*\*コンテキスト崩壊\*\*：コードベースがエージェントの有効コンテキストウィンドウを超えると、過去の決定を忘れて矛盾した実装をするケース。第三は\*\*検証不能な出力\*\*：明示的な受け入れ基準がなく、エージェントのコードが「正しい」かどうかの判断基準がないケース。



2025年9月にはFast Companyが「vibe codingのハングオーバー」を報告し、シニアエンジニアたちがAI生成コードによる「開発地獄」を経験していることが明らかになった。



\---



\## 2. 最初の対応：AGENTS.md / CLAUDE.md の登場



この「毎回同じ前提を説明する手間」の問題に対し、ツールベンダー側が構造的な解決策を提供した。



OpenAIのCodexは「リポジトリ内に置いたAGENTS.mdファイルをCodexが自動で読む」機能を実装した。これはREADME.mdに近いテキストファイルで、コードベースのナビゲーション方法・テスト実行コマンド・プロジェクトの標準的慣行などをCodexに伝える目的で使われる。



AGENTS.mdはOpenAI Codex、Google Julesのような各社の取り組みから生まれ、2025年12月にはLinux Foundation傘下の\*\*Agentic AI Foundation（AAIF）\*\*に移管され、オープン標準として維持されるようになった。



Codexはプロジェクトルートから現在のディレクトリまでを下向きに走査し、各ディレクトリのAGENTS.mdを連結して読み込む。後から見つかったファイルが先の指示をオーバーライドする設計になっており、モノレポでのサブディレクトリ固有の上書きに利用できる。



ただしこの段階では「エージェントへの文脈提供」が解決されただけで、スコープ制御の問題は残存していた。



\---



\## 3. スペック駆動の発生：「仕様を先に書く」という発想



AGENTS.mdが「文脈の永続化」を解決した後、次の問題は「何を作るかの定義が実装前に存在しない」ことだった。



スペック駆動開発（SDD）は、仕様書をソース・オブ・トゥルースとして扱い、コードを派生・検証済みのセカンダリ成果物として位置づける。「写真共有機能を追加して」という指示ではなく、「ユーザーはJPEGまたはPNGを10MBまでアップロードできる。写真はS3にユーザーIDプレフィックス付きで保存される」という形式の仕様に変換してから実装させる。



この発想の知的背景としてはTDD（テスト駆動開発）とBDD（振る舞い駆動開発）が参照された。



Thoughtworksの分析によれば、SDDは技術的には「ソフトウェアの外部ふるまいを明示的に定義する」ものであり、入出力マッピング・事前条件/事後条件などを含む。単なる製品要求書（PRD）を超えており、より構造化されたプロンプトと明示的な技術的制約を持つ。



良いSpec Cardが定義する6要素は、アウトカム・スコープ境界・制約・事前決定事項・タスク分割・検証基準とされている。



\---



\## 4. PROJECT\_CONTRACT.md の発生：タスク横断の制約



個別タスクのSpecだけでは不足することが判明した。「このリポジトリで\*\*絶対にやってはいけないこと\*\*」のタスク横断的な定義が必要になった。



GitHub Spec KitやBMADの「constitution（憲法）」ファイルがこれに相当する。典型的な内容は「このシステムはTypeScript strictモードを使用しなければならない。このシステムはテストカバレッジを下げるPRを拒否しなければならない」といったEARS形式の宣言になる。これにより、エージェントは要件を読んでコードを生成し、それを検証するテストも書けるようになる（スコープについての推測なし）。



このConstitutionの有効性には、記述の具体性が決定的に重要だという認識が広まった。



\- Bad例：「高品質なコードを書くこと」

\- Good例：「ファイル出力は`--out-root`以下に限定される。外部に出るパスは失敗しなければならない」

\## 5. ワークフロー分岐の発生：リスクによる処理の分離



エージェントが自律的に動ける作業と人間が必ず確認すべき作業の混在が問題化した。



受け入れ基準は実行可能（Executable）でなければならない。実行できないものは意見であって基準ではない。また二値（Binary）である必要がある——「適度に速い」や「コードがきれい」は基準ではない。独立（Independent）でもある必要があり、「テスト通過かつカバレッジ85%かつネットワーク呼び出しなし」を一つの基準にまとめるべきでない（何が失敗したか分からなくなる）。



ワークフロー分岐の実装では、`risk`フィールドによるlow-risk / high-riskの分離が典型的なパターンとなった：



```text

low-risk：  Planner → Implementer → 自動検証 → 完了

high-risk： Planner → \[人間確認] → Implementer → Reviewer → \[人間確認] → 完了

```



ファイル削除・サブプロセス実行・ネットワーク送信が high-risk の典型例。



\---



\## 6. エージェント分離の発生：役割の専門化



単一エージェントへの全役割集中が「自分で実装して自分でレビューする」という構造的問題を生むことが認識された。



AgentMesh（2025年）はPlanner・Coder・Debugger・Reviewerの4専門エージェントを分離する構造を提案している。Plannerがユーザー要求をサブタスクに分解し、Coderが実装し、Debuggerがテスト・修正し、Reviewerが要件との照合と品質評価を行う。ChatDevなどの研究では、このような専門エージェントの分業が単一エージェントよりも複雑タスクで優れることが示唆されている。



この構造においては各エージェントの「読んでよいファイル」「変更してよいファイル」を明示的に制限することで役割の越境を防ぐ設計が定石となった。

\## 7. 現在の位置：ツールの乱立と収束



SDDのツール景観は2025年7月のGitHub Spec Kit発表からAWS Kiroのリリースを経て2026年初頭にかけて急速に拡大した。



主要ツールの位置づけ：



| ツール | 規模・特性 |

|---|---|

| GitHub Spec Kit | オープンソース・constitution → spec → plan → tasks → implementation の完全フロー |

| AWS Kiro | IDE統合型・"prototype to production"を掲げる商用環境 |

| cc-sdd | 17スキル × 8エージェント対応の中規模マルチエージェントハーネス |

| OpenSpec | 軽量・ブラウンフィールド（既存コードベース改修）向け |

| BMAD | 多役割・大規模プロセス（Analyst / PM / Architect / Developer / QA） |



ただし有効性については批判的研究も出ている。ETH ZurichとLogicStar.aiによる研究（arXiv:2602.11988、2026年2月）は、複数のコーディングエージェントとLLMを検証した結果、コンテキストファイルはリポジトリコンテキストなしの場合と比べてタスク成功率を下げる傾向があり、推論コストを20%以上増加させることを示した。



この研究の解釈として、LLM生成のコンテキストファイルはパフォーマンスを低下させる一方、開発者が書いた最小限のコンテキストファイルには小さなポジティブ効果があることが示された。

\## 8. まとめ、そして将来像



\*\*これまでの流れを一言で言えば：\*\*



LLMエージェントへの作業委譲が進むにつれ、「曖昧な指示」を「検証可能な構造」に変換する必要性が段階的に明確化してきた歴史。



| 時期 | 解決した問題 | 生まれた概念 |

|---|---|---|

| 2022〜2023年 | 毎回同じ前提を伝える手間 | AGENTS.md / CLAUDE.md |

| 2023〜2024年 | スコープ逸脱・完了定義の不在 | Spec Card（Must/Must not/Done） |

| 2024年〜 | タスク横断の制約定義 | PROJECT\_CONTRACT.md / Constitution |

| 2024〜2025年 | 作業種別によるリスク分離 | ワークフロー分岐 |

| 2025年〜 | 役割混在による品質劣化 | エージェント分離（Planner/Implementer/Reviewer） |



\*\*現在見えている新しい手法の芽：\*\*



2025〜2026年に浮上している次のパラダイムは\*\*Context Engineering（コンテキストエンジニアリング）\*\*と呼ばれている。



プロンプトエンジニアリングが「どう指示するか」に注目するのに対し、コンテキストエンジニアリングは「モデルが動作する情報環境全体」——メモリ・検索ドキュメント・ツール定義・会話履歴——を設計することに注目する。Patrick Deboisは2024年のAI Engineer Londonで「Context is the new Code」という概念を提唱し、コンテキスト開発ライフサイクル（CDLC）がソフトウェア開発ライフサイクル（SDLC）と対応するという枠組みを示した。



arXiv論文「Spec-Driven Development: From Code to Contract in the Age of AI」（2026年2月）は、伝統的な仕様書が人間に読まれるのに対し、SDD仕様書は\*\*検証ゲートとして実行される\*\*という核心的な区別を示している。



この方向性は「仕様書が静的ドキュメントから実行可能な制約へ」という移行であり、コードではなく\*\*コンテキストと仕様の設計こそが中心的なエンジニアリング作業になる\*\*という将来像につながっている。
